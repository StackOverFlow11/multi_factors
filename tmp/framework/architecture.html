1设计原则（架构的脊梁）

所有结构决定都从这 5 条推出来。看不懂某个设计为什么这么分时，回到这里对照。

#原则含义 & 为什么
1回测即实盘
backtest = live		 边界做 adapter、中间共享同一份定义和数据。factor / alpha / portfolio 层一行不改地两边复用，差异只压在系统"边缘"。这是"渐进到实盘"能省力的根本。
2防未来函数优先
lookahead-safe by design		 宁可慢、宁可多写一层，也要让"用到未来信息"在架构上难以发生。截面里未来函数有 4 个隐蔽来源（见 §8），每一个都对应一层防线。
3分层解耦
separation of concerns		 factor 不碰数据源、strategy 不碰下单。每层只认上一层的接口。坏味道：策略里直接 tushare.get(...)。
4渐进 / 不过度设计
progressive		 先打通最细一条链路（MVP），按需加层。新手头号坑是一上来就搭完整 Feature Store。见 §11 路线图。
5透明可控
transparent over black-box		 核心（因子/合成/组合/截面回测）自建——内核很短、完全可读。把 qlib 这类黑盒框架当参照而非地基。

为什么 vnpy 从"主干"退为"下单通道"
vnpy / backtrader 是逐标的、事件驱动，为择时(CTA)而生。截面是"每个调仓日把全市场几百只股票一起排序"——这是它们的死穴。所以回测主干换成 pandas/polars 面板 + 自写调仓回测，vnpy 只在实盘阶段当下单网关（也可换 QMT / 券商 API）。

3核心数据结构：Panel 面板

截面框架里一切都是面板：行情、因子、得分、权重、收益，全是同一个形状——MultiIndex (date, symbol) 的 DataFrame。这是与择时最大的结构差异（择时是单标的一维时间序列）。

# 取某一天的"截面"（横切一刀）= 当天所有股票的因子，用于排序选股
panel.loc["2026-05-29"]        # -> 一个截面 DataFrame
# 取某只股票的"时序"（纵切）= 因子在时间上滚动算的来源
panel.xs("600519", level="symbol")
时序 vs 截面，不矛盾
因子计算是时序的（动量=过去20日收益、波动=过去60日std），但决策是截面的（每个时点横向比较、挑分高的）。"正统多因子"= 用时序算特征、在截面上做选择。你说的"factors/portfolio 有点择时性质"指的就是前半句，没问题。

4标准多因子工作流

截面的工作流和择时很不一样：不是写完策略直接回测，而是先验证因子有没有用（IC/分层），有效才进组合。

① 单因子compute

② 预处理去极值/标准化/中性化
→
③ 因子检验alphalens: IC / 分层
→
④ 多因子合成alpha
→
⑤ 组合构建portfolio

⑦ 实盘execution

③ 是择时框架里没有的关键关卡：先用 alphalens 看因子的 IC（信息系数，因子值与下期收益的截面相关）和分层单调性，淘汰无效因子，再花力气进组合回测。省时间、也防过拟合。

5各层详解 + 接口签名

接口先定签名、不写实现——这就是"分层解耦"落到代码上的样子。每层只依赖下一层的抽象接口。

5.1 数据层 data/

采集 → 清洗（复权 + 按披露日对齐财务数据）→ 存成面板 parquet。feed 是边缘（回测=历史、实盘=实时），但对上层暴露同一接口。

# data/feed/base.py
class DataFeed(ABC):
    def get_bars(self, symbols, start, end, freq) -> DataFrame: ...   # 返回 panel

# data/store/panel_store.py  —— 按 date 分区，DuckDB 直接 SQL 查
class PanelStore:
    def write(self, name, df): ...
    def read(self, name, start, end, symbols=None) -> DataFrame: ...

5.2 域层 universe/ 截面新增

截面独有、且是最容易被忽略的偏差来源。两个职责：① 给出某日的成分股（point-in-time，不能用今天的成分回测历史）；② 过滤"看得到买不到"的标的。

# universe/base.py
class Universe(ABC):
    def members(self, date) -> list[str]: ...    # PIT 成分股（沪深300/中证500/全A）
    def tradable(self, date) -> list[str]: ...   # 排除 停牌/涨跌停/ST/上市未满N天

5.3 因子层 factors/ 自建核心

拆三块：compute（定义+计算）、process（截面预处理三件套）、store（落盘，见 §7）。因子定义只写一次，batch（回测）和 incremental（实盘）是它的两种执行方式。

# factors/base.py
class Factor(ABC):
    name: str; deps: list[str]; params: dict
    def compute(self, panel) -> Series: ...         # 输入面板 -> 因子值, batch（回测）
    def update(self, state, bar) -> float: ...      # 增量更新, 实盘（可选, 见 §7）

# factors/process/pipeline.py —— 每个截面（按 date 分组）依次做：
def winsorize(s, method="mad", n=3)  -> Series: ...   # 去极值
def standardize(s, method="zscore")  -> Series: ...   # 标准化（或 rank）
def neutralize(s, industry, mktcap)  -> Series: ...   # 中性化：回归取残差，去掉行业+市值暴露

5.4 合成层 alpha/ 截面新增独立成层

把多个处理后的因子合成一个综合得分。独立成层、不并入 factors——合成要用未来收益拟合权重，必须和"永不碰未来收益"的因子层隔开（守住防未来函数边界，见 §7）。从等权起步，逐步到 IC 加权、线性回归，将来可上树模型（这时才需要 ML）。

# alpha/base.py
class AlphaModel(ABC):
    def fit(self, factors, fwd_returns): ...
    def predict(self, factors_today) -> Series: ...   # -> 每只股票的综合得分
# 实现: EqualWeight -> ICWeight -> LinearReg -> (LightGBM 以后)

5.5 组合层 portfolio/ 截面新增（含 risk 约束）

得分 → 目标权重。风控的"事前约束"并入这一层（约束本就是组合构建的 constraints）：最大权重、行业中性、换手上限。要考虑当前持仓以控制换手（成本敏感）。运行期风控（实盘熔断/监控）不在这层，归 runtime/live。

# portfolio/construct.py
class PortfolioConstructor(ABC):
    def build(self, scores, current_w) -> Series: ...   # -> 目标权重
# 实现: TopN 等权 -> 分位数分层 -> 均值方差优化(cvxpy/riskfolio)

# portfolio/risk.py —— 事前约束（构建期）
def apply_constraints(w, industry, max_w, turnover_cap) -> Series: ...

5.6 运行层 runtime/（边缘 · 合并原 backtest + execution）

回测和实盘是同一层的两套实现，各含两个角色：driver（回测=遍历历史调仓日 / 实盘=每日定时触发）+ execution（回测=模拟撮合成本 / 实盘=券商下单）。这是"回测即实盘"落到代码上的样子。

driver —— 截面调仓驱动（内核很短，这就是不需要重型引擎的原因）

# runtime/backtest/driver.py   （实盘版: 把循环换成"每日定时触发一次"）
def run(scores_panel, universe, constructor, execution, rebalance="M"):
    nav = 1.0
    for date in rebalance_dates(rebalance):
        uni    = universe.tradable(date)              # 当日可交易池
        score  = scores_panel.loc[date, uni]
        w      = constructor.build(score, current_w)  # 目标权重
        execution.rebalance_to(w)                     # 交给执行角色（回测/实盘同接口）
        nav   *= 1 + execution.last_return()
        current_w = w
    return nav_series

execution —— 同接口、两实现

# runtime/execution.py
class Execution(ABC):
    def rebalance_to(self, target_w): ...    # 调到目标组合
    def positions(self) -> Series: ...
# backtest/ -> SimExecution(撮合/成本)   live/ -> LiveExecution(vnpy gateway / QMT)

6投研↔实盘 seam（Ports & Adapters）

投研和实盘的差异本质在系统"边缘"，集中在 data/feed 与 runtime 两层。截面因为低频批处理，seam 比择时更薄——时钟差异退化为"定时跑一次"：

边界（Port）回测实现实盘实现上层是否感知
data/feed历史面板回放当日最新面板❌
runtime/driver遍历历史调仓日（模拟时钟）每日定时触发（真实时钟）❌
runtime/execution撮合 + 成本模型券商/QMT 下单❌
落地提醒：execution 抽象别和 vnpy 打架
如果实盘用 vnpy 当下单通道，vnpy 的 gateway 本身就是一套 adapter。你的 Execution 接口要薄薄地包一层就好，不要重写 vnpy 已有的东西。抽象的价值在于让 portfolio 以上不感知下单细节，不在于复刻网关。

7因子层：Feature Store 与一致性

因子拆"计算 / 存储 / 元数据"是对的（这是 ML 工程的 Feature Store 模式）。但真正的难点不在存储，在计算的一致性：

头号验收项：batch ≡ incremental
同一因子，回测是 batch 算（整列 pandas），实盘是 incremental 算（每根新 bar 增量更新）。两条路必须算出完全一致的值，否则回测在"撒谎"，实盘行为对不上。这在 ML 里叫训练-服务偏移 (training-serving skew)，是 Feature Store 存在的头号理由。
对策：因子定义只写一次；batch / incremental 是两种执行方式；用一条单元测试锁死"两者在同一段数据上输出相等"。
子层职责关键风险 / 要求何时做
compute 计算定义 + batch/incremental 两种执行一致性（上面那条）MVP
store 存储因子值落盘、缓存，避免重算必须带时间戳(point-in-time)；回测/实盘读同一份 → 防偏移MVP(简版)
registry 元数据因子名、依赖、参数、版本因子多了才需要，别过早做后期

MVP 阶段：因子就是纯函数 def factor(panel) -> Series，batch 算、存 parquet 当缓存，元数据用文件名/简单 dict。因子数 >20 或开始实盘需要增量时，再上正式的 registry + 增量引擎。
